## Enrichment Bug Fix Session (Apr 11, 2026) — CRITICAL

### The Core Bug
The `enrich-trym-leads.js` script was calling Supabase PATCH with `Prefer: representation` which returns HTTP 204 No Content — the update silently FAILS. The script logged "✓" for every lead but wrote NOTHING to the database. ~257 leads worth of "success" logs were completely false.

**Fix**: Change header to `Prefer: return=representation` so Supabase returns the updated row body and we can verify success.

### Second Bug (Double JSON Serialization)
`photos` column was being `JSON.stringify()`'d in the scrape step, then the whole object was re-stringified again before the fetch call — resulting in double-encoded JSON. Caused `[" " must introduce explicitly-specified array dimensions` PostgreSQL errors.

**Fix**: Don't re-stringify already-stringified fields.

### Third Bug (execSync Shell Escaping)
Using `execSync` with shell curl for JSON payloads corrupts the JSON (quotes get mangled by the shell). Node `fetch` API is more reliable.

**Fix**: Replace `execSync` curl calls with `fetch()` in the `supabaseUpdate` function.

### Fourth Bug (Wrong Database Query)
The `.env` in mission-control-next had the correct `SUPABASE_URL` but `curl -s 'https://zexumnlvkrjryvzrlavp.supabase.co/rest/v1/trym_leads?limit=5'` returned 0 rows — yet Mission Control showed 3,000 leads. This was a red herring: the Supabase REST API with anonymous key returns 0 rows for `trym_leads` (RLS policy blocking), but the service role key (used by Next.js API routes + Mission Control's SSR) DOES return rows.
